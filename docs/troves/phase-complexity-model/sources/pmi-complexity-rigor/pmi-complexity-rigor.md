# Using Project Complexity Determinations to Establish Required Levels of Project Rigor

PMI paper. Also published via OSTI.gov (DOE Office of Scientific and Technical Information).

## Core Concept

The level of a project's complexity determines what level of project rigor is needed to effectively complete the project and meet necessary goals. Uses technical competencies and skills to determine project complexity and establish required levels of project rigor.

## Graded Approach

The paper presents a **graded approach** that scales the following process areas proportional to project complexity determination (low, medium, high):

1. **Team formation** — extent and formality
2. **Planning documentation rigor** — level of detail
3. **Stakeholder communication rigor** — frequency and formality
4. **Requirements management rigor** — traceability depth
5. **Acquisition planning** — procurement oversight
6. **Risk management** — assessment depth and response planning
7. **Executing** — oversight and control intensity
8. **Monitoring and controlling** — reporting frequency and metrics

## Implementation

The complexity determination process was integrated into the company's project management procedure by specifically defining the **minimum process group rigor for each complexity level** (low, medium, high) with required steps/activities.

## Key Insight for Swain

This is the most direct precedent for what swain needs: a formal mapping from a scored complexity level to concrete process rigor decisions. The graded approach says "low complexity → light process, high complexity → full process" — exactly the pattern the phase complexity model proposes for ceremony, model selection, manual tests, and autonomy.

## Complementary: Kathleen B. Haas Complexity Model

PMI also references Haas's "Project Complexity Model" which distinguishes between complicated and complex:
- **Complicated**: the business problem might be difficult to define but the solution is discoverable
- **Complex**: business process relationships are non-linear with multiple feedback loops
- Recommends: "Do not be tempted to apply more rigor than necessary. Continually verify that the project is building the minimum viable solution. Use the motto: 'Barely sufficient is enough to move forward.'"
