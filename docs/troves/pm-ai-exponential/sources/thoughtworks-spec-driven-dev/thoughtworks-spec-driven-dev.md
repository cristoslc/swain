# Spec-driven development

**Author:** Liu Shangqi, Technology Director, APAC Region, Thoughtworks
**Date:** December 4, 2025

Spec-driven development (SDD) is a development paradigm that uses well-crafted software requirement specifications as prompts, aided by AI coding agents, to generate executable code.

## Competing interpretations

There are different opinions within the industry about what a spec is and its role in SDD:

- **Radical view:** Code is a byproduct. Specs are the sole source of truth that needs maintenance. Code is an intermediate product between requirements and compiled binaries.
- **Conservative view (author's position):** Specs are merely elements that drive code generation, as in test-driven development. Executable code remains the source of truth.

## Context of emergence

When the industry first began using generative AI to produce code, it found that code generated by setting technical specifications (technology stack, architectural style, coding style) was of higher quality compared to plain-text requirements. As AI coding tools improved, it became easier to bring functional requirements into the picture — but we can't rely on functional requirements alone: we need to pay attention to the technical details.

The latest AI coding agents generally separate the planning and implementation phases. The planning phase focuses on understanding requirements, designing constraints and curating prompts. This is essentially the process of creating the specification.

## What is a spec?

A specification is more than just a PRD. Even simply applying a more structured prompt and more explicit technical constraints can produce better code than a plain PRD.

Technically, a specification should explicitly define the external behavior of the target software — things like input/output mappings, preconditions/postconditions, invariants, constraints, interface types, integration contracts and sequential logic/state machines.

### What makes a good spec?

- Use domain-oriented ubiquitous language to describe business intent rather than tech-bound implementations
- Clear structure with common style (Given/When/Then)
- Completeness yet conciseness — cover the critical path without enumerating all cases
- Clarity and determinism — clear specifications reduce model hallucinations
- Structured inputs and outputs significantly improve reasoning performance
- Machine-readable specs remain essential in the LLM era

## SDD in practice

Tools like Amazon Kiro and GitHub Spec-Kit offer predefined workflows. With methodologically neutral tools like Cursor or Claude Code, you find your own workflow.

The core: separate design and implementation phases. Requirements are analyzed using an AI coding agent, generating design and implementation plans formalized into Markdown files. Reviewing and validating these specifications is iterative and requires a human in the loop.

## Is SDD just a return to waterfall?

The author argues this time is different:

- Waterfall's problem was excessively long feedback cycles and disconnect between design and implementation
- Vibe coding's problem is being too fast, spontaneous and haphazard — overlooking good engineering practices, producing unmaintainable code
- SDD brings serious requirements analysis, prudent software design, necessary architectural constraints, and human-in-the-loop governance — providing shorter, more effective feedback loops than pure vibe coding

## Challenges and risks

- Lack of consensus on the "correct" SDD workflow
- Code generation from spec to LLM isn't deterministic — poses challenges for upgrades and maintenance
- Spec drift and hallucination are inherently difficult to avoid
- Still need highly deterministic CI/CD practices to ensure software quality
- Over-formalized specs can cause trouble and slow down feedback cycles — same risk as early waterfall

> "Spec-driven development remains an emerging practice as 2025 draws to a close; we're likely to see even more change in 2026."
