# Retro: v0.10.0-alpha Release

**Scope:** Release v0.10.0-alpha
**Date:** 2026-03-21

## Summary

This release introduced the swain-roadmap skill and `chart.sh roadmap --cli` renderer, updated the swain-release changelog format to use four fixed sections (Features, Roadmap, Research, Supporting Changes), and added session facilitation planning artifacts.

## Reflection

### What went well

**Thematic changelog with operator pause.** The release workflow pauses after proposing the changelog so the operator can adjust bucketing before committing. This caught a misclassification (session facilitation artifacts listed as "features" when they were roadmap additions, iTerm fix listed as a feature when it was a supporting change). The pause-and-adjust pattern is the right default for changelogs — the agent proposes, the operator corrects.

### What to change

**Move to a Jinja2 template for changelog.** The four-section structure (Features, Roadmap, Research, Supporting Changes) is currently prose instructions in the skill. Encoding it as a Jinja2 template would make the semantics machine-readable and reduce the agent's interpretive burden — the template defines the sections, the agent fills slots.

### Patterns

**Agent not reliably bucketing changes.** Across iterations, the agent consistently struggled with classifying changes into the right section. Planning artifacts (new visions, initiatives, epics, specs) were bucketed as "features" when they represent future work, not shipped capability. Bug fixes and cross-reference enrichment were bucketed as features when they're supporting changes. This is a semantic distinction the agent doesn't naturally make — it sees "new file created" and infers "feature." The Jinja2 template approach may help by making the categories structural rather than interpretive.
