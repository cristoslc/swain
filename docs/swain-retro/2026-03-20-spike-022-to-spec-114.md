## Reflection

### What went well

**Retro-as-artifact was gamechanging.** Starting from a saved retro document meant the failure details (EPIC-038's missing enrichment fields) were permanently available, not trapped in fragile agent context. Every re-read surfaced new insights — the retro was referenced at least 5 times during the session, and each reading revealed something the previous analysis had missed (rebase vs checkout vs merge as the actual mechanism). The permanent artifact pattern made iterative deepening possible.

### What was surprising

**The agent vastly overcomplicated the initial solution.** The first-pass analysis produced a three-layer mitigation strategy (pre-dispatch heuristic, CAS file-overlap check, serialized test gate), framed the problem as TOCTOU, proposed merge queues and commit queues, and explored local serialization mechanisms. The actual fix was "use merge instead of rebase" — something git already does. It took operator pushback ("this must be a solved problem", "are we reinventing git?", "how does GitHub do it?") to cut through the overengineering.

**The agent repeatedly missed important details, identified wrong problems, and drew wrong conclusions.** Specific misses:

1. Claimed the root cause was "no recovery path for concurrent completion" without evidence
2. Assumed the operator manually resolved the EPIC-038 failure — the operator didn't
3. Didn't read ADR-005 (the existing workflow) until the operator asked "why was cherry-pick being used?" — should have been the first thing read
4. Framed the problem as TOCTOU when the most plausible failure was rebase auto-resolving
5. Proposed testing as the fix when the actual issue was data loss during integration
6. Built a three-layer strategy before checking whether git's merge handles it
7. Made confident claims in the ADR context section that couldn't be proven

### What would change

**Read before reasoning.** Before analyzing a problem or proposing a solution, read the existing artifacts — not just the immediate one, but linked artifacts (parent EPICs, related ADRs, referenced SPECs, troves). If we'd read ADR-005 at the start, we'd have seen "rebase + push HEAD:main" immediately and skipped the TOCTOU framing entirely.

**Evidence before conclusions — universally.** Every claim must be grounded in something observable. If you can't point to the evidence, say "I don't know" or "this is speculation." This is swain-search's "sources before synthesis" principle applied to all agent activity: debugging, root cause analysis, solution design, implementation plans, releases, ADR context sections. The agent should build a visible evidence basis (an index of artifact IDs, commit hashes, file blobs) before every action, persisted for retrospective analysis.

### Patterns observed

**"Reaching conclusions before verifying premises, then building on unverified conclusions."** This is the core anti-pattern from this session. Each layer of analysis inherited the errors of the previous layer. The operator's corrections were all variants of "wait, can you prove that?" and "does that match what actually happened?" The fix is structural: evidence index before every action, visible to the operator, persisted for review.

**Operator as simplifier, agent as complicator.** The most productive moments were when the operator cut through complexity: "this must be a solved problem", "we can't assume GitHub", "serialized merge into main should also solve it", "the issue was coordination resulting in an overwrite". The agent's instinct was to build new infrastructure; the operator's instinct was to use existing primitives correctly.

**The spike investigation was still valuable despite overengineering.** SPIKE-022's broad investigation ruled out complex approaches and built a knowledge base (trove with 18 sources). The three-layer strategy was wrong, but the research that produced it confirmed the simpler answer was correct. Exploration isn't wasted even when the conclusion is "do the simple thing."

**We hit the SPIKE-022 failure class during implementation.** The ref-update agent (swa-lv26) conflicted with the swain-sync agent (swa-ol3y) on SKILL.md — two parallel worktree agents modifying the same file. The merge resolved cleanly by keeping HEAD (which already had the correct changes). This was both ironic and validating — the problem we were fixing manifested while we were fixing it, and the merge-based approach handled it correctly.
