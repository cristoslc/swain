## Reflection

### What went well

**Autonomous decomposition and parallel execution.** Swain handled epic creation, spec writing, parallel worktree agents, and artifact lifecycle transitions (supersession, specwatch-ignore, index rebuild) with minimal operator steering. The operator could focus on design decisions while agents handled file operations.

**Rapid visual prototyping loop.** The cycle of "generate → render → screenshot → feedback → iterate" was effective for converging on the right visual design. Eight iterations moved from a broken Gantt to a working Eisenhower quadrant chart with side-by-side PNG+legend layout. Each iteration was informed by the operator's Gestalt principles feedback (proximity, similarity, continuity, figure/ground).

**Design principle capture.** The "descriptive not prescriptive" principle was captured mid-session and embedded in the epic — preventing the roadmap from becoming a manually-edited plan instead of a computed projection.

**Bug fix velocity.** gh#80 went from issue read → spec → worktree agent → cherry-pick → push → close in under 3 minutes.

### What was surprising

**Mermaid's limitations are severe.** quadrantChart has no legend, no label control, labels clip at boundaries, and GitHub's Mermaid version doesn't even render it. The workaround (pre-rendered PNG via mmdc) was discovered through systematic elimination of CSS approaches (flex, float, grid all failed in Typora). This took ~45 minutes of trial and error that could have been avoided with upfront research.

**Specs grew reactively instead of architecturally.** Each spec was a response to the last screenshot — "fix the labels" (104), "fix the label strategy" (105), "fix the flowchart" (106). None addressed the root cause (no unified data model). The operator caught this: "the refactor should be blocking all the renderers. do we need to go back to the epic objectives and rebuild the specs?" This led to a clean decomposition but cost ~30 minutes of spec churn.

**Worktree agent merge conflicts are silent.** SPEC-107 and SPEC-108 both modified roadmap.py. The SPEC-108 agent reported success with 12 passing tests, but after checkout into main, the enrichment fields were missing. The agent tested its own worktree copy, not the merged result. This is a systemic risk with parallel worktree agents on overlapping files.

### What would change

**Research before rendering.** Should have run swain-search on Mermaid quadrantChart capabilities *before* the first render attempt. Would have discovered the label/legend limitations upfront and gone straight to the short-ID+PNG approach.

**Architecture-first spec decomposition.** The exploratory implementation was valuable for learning what the roadmap needs, but the specs should have been written from the architecture (data model → templates → renderers) not from the output (chart → tables → labels). The supersession was correct but expensive — 5 specs created, iterated on, then thrown away.

**Verify agent deliverables after merge, not in worktree.** When two agents modify the same file, the later checkout overwrites the earlier. Must grep for key deliverables and re-run tests in main context before claiming delivery.

**Sync latency breaks flow.** Every /swain-sync call was a 60-90 second context break. The operator identified this as a standalone problem (SPEC-113). Background-by-default or a shell script implementation would help.

### Patterns observed

**Operator as design critic, agent as implementer.** The most productive pattern was: agent proposes visual output → operator critiques with Gestalt principles → agent iterates. The operator never wrote code but shaped every visual decision.

**"CHATONLY>" signals design discussion.** When the operator prefixed with CHATONLY>, they wanted to think through design decisions without triggering implementation. This was effective for the Eisenhower mapping, the descriptive-not-prescriptive principle, and the markdown-vs-HTML surface decision.

**Exploratory implementation has value even when superseded.** The 5 superseded specs produced working code that validated the feature's requirements. The architectural decomposition was only possible because the exploratory work revealed the real structure (data model + templates + renderers). Prototyping first, then restructuring, is a valid pattern — but the specs should have been labeled as exploratory from the start.

**The Eisenhower framing was operator-driven and highly effective.** The agent initially produced flat priority lists. The operator pushed toward Eisenhower (important × urgent) which immediately revealed that the scoring model needed better differentiation (leading to SPEC-107 sort-order). The framework also naturally surfaced decision types (activate/drop, decompose, complete) which became the "Needs" column.
