# Output Shaping for Chat

## Problem Statement

Agent responses generated by the runtime are optimized for terminal interfaces — verbose, unstructured, and often too long for chat windows. When these responses are relayed to the operator via the chat adapter, they create cognitive overload and reduce signal-to-noise ratio in the chat interface. Post-processing every response through an LLM would introduce significant latency.

## Desired Outcomes

Operators receive agent responses that are concise, well-formatted, and appropriate for chat consumption. The project bridge achieves this by injecting formatting and length instructions into the agent's system/user prompts at session genesis, ensuring the agent generates chat-optimized output natively without introducing per-response latency.

## External Behavior

**Inputs:**
- Operator configuration for output shaping preferences (length, format, tone) from project bridge config.
- Agent session genesis event from the runtime adapter.

**Outputs:**
- System/user prompt instructions injected into the agent's context at session start, shaping all subsequent output.
- Agent output (already shaped by following the injected instructions) forwarded unchanged to the chat adapter.

**Constraints:**
- Shaping instructions are injected once at session genesis — zero per-response latency.
- Shaping is configurable per project bridge via config file.
- Code blocks and critical formatting (e.g., tables, lists) are preserved by instructing the agent to maintain them.

## Acceptance Criteria

1. **Given** a project bridge with output shaping enabled, **when** a session is spawned, **then** the project bridge injects system/user prompt instructions that specify the configured length constraints (e.g., "keep responses under 200 words unless code or detailed explanation is requested").

2. **Given** a project bridge with markdown enforcement, **when** a session is spawned, **then** the project bridge injects instructions that tell the agent to "always format responses as valid markdown".

3. **Given** a project bridge with length preferences configured (short/medium/long), **when** a session is spawned, **then** the project bridge injects instructions that tell the agent the preferred response length (e.g., "prefer concise responses — 2-3 sentences unless the operator asks for detail").

4. **Given** a project bridge with formatting rules, **when** a session is spawned, **then** the project bridge injects instructions that tell the agent to preserve code blocks, tables, and lists while keeping prose concise.

5. **Given** output shaping is disabled in config, **when** a session is spawned, **then** the project bridge does not inject any shaping instructions and the agent responds with its default behavior.

6. **Given** an active session with output shaping enabled, **when** the agent responds, **then** the response is already shaped (no post-processing by the project bridge).

## Scope & Constraints

**In scope:**
- Configuration schema for output shaping rules (length limits, markdown enforcement, length preference, tone).
- Prompt injection logic in project bridge at session genesis — injecting system/user instructions that shape agent output.
- Runtime adapter integration point for injecting instructions when spawning the agent.
- Tests verifying instructions are injected correctly and agent output is shaped.

**Out of scope:**
- Post-processing or summarizing agent output after it's generated (adds latency).
- Chat adapter changes (adapter receives shaped output as-is).
- User-facing commands to toggle shaping mid-session (future enhancement).
- Modifying the agent's core system prompt permanently — instructions are session-scoped.

## Implementation Approach

1. **Config schema:** Add `output_shaping` section to project bridge config with fields: `enabled`, `max_length`, `markdown_only`, `length_preference`, `tone`.

2. **Prompt template:** Create a prompt template system in the project bridge that generates system/user instructions from the config. Example:
   ```
   "You are an AI assistant communicating via a chat interface. Follow these guidelines:
   - Keep responses concise: aim for {length_preference} (short=2-3 sentences, medium=1 paragraph, long=detailed as needed)
   - Always use valid markdown formatting
   - Preserve code blocks, tables, and lists exactly
   - Maximum {max_length} characters unless operator requests more detail"
   ```

3. **Genesis injection:** Wire the prompt template into the project bridge's session spawn flow — inject the generated instructions into the runtime adapter's spawn command as system/user prompt arguments.

4. **Runtime adapter integration:** The runtime adapter accepts system/user prompt overrides at spawn time and passes them to the agent.

5. **Tests:** Write integration tests verifying:
   - Instructions are correctly generated from config
   - Instructions are passed to runtime adapter at session spawn
   - Agent output follows the injected instructions (behavioral test)
