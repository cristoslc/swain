# Output Shaping for Chat

## Problem Statement

Agent responses generated by the runtime are optimized for terminal interfaces — verbose, unstructured, and often too long for chat windows. When these responses are relayed to the operator via the chat adapter, they create cognitive overload and reduce signal-to-noise ratio in the chat interface.

## Desired Outcomes

Operators receive agent responses that are concise, well-formatted, and appropriate for chat consumption. The project bridge shapes output without modifying the agent's core behavior — it acts as a formatting layer between the runtime and the chat adapter.

## External Behavior

**Inputs:**
- Raw agent output (text, code blocks, structured data) from the runtime adapter via NDJSON `agent_output` events.

**Outputs:**
- Shaped output sent to the chat adapter as `message` events with appropriate formatting constraints applied.

**Constraints:**
- Shaping is configurable per project bridge via config file.
- Shaping rules are applied after agent output is received, before forwarding to chat adapter.
- Code blocks and critical formatting (e.g., tables, lists) are preserved.

## Acceptance Criteria

1. **Given** a project bridge with output shaping enabled, **when** the agent emits verbose output, **then** the project bridge truncates or summarizes output exceeding a configurable character limit.

2. **Given** a project bridge with markdown enforcement, **when** the agent emits unformatted text, **then** the project bridge ensures the output is valid markdown before sending to chat.

3. **Given** a project bridge with length preferences configured, **when** the agent emits a response, **then** the project bridge applies the configured length constraint (short, medium, long) to the output.

4. **Given** a project bridge with formatting rules, **when** the agent emits output, **then** code blocks, tables, and lists are preserved while prose is shaped according to config.

5. **Given** output shaping is disabled in config, **when** the agent emits output, **then** the project bridge forwards it unchanged to the chat adapter.

## Scope & Constraints

**In scope:**
- Configuration schema for output shaping rules (length limits, markdown enforcement, length preference).
- Shaping logic in project bridge between runtime adapter output and chat adapter input.
- Tests for each shaping rule.

**Out of scope:**
- Modifying agent behavior or prompts directly.
- Chat adapter changes (adapter receives shaped output as-is).
- User-facing commands to toggle shaping mid-session (future enhancement).

## Implementation Approach

1. **Config schema:** Add `output_shaping` section to project bridge config with fields: `enabled`, `max_length`, `markdown_only`, `length_preference`.

2. **Shaping pipeline:** Create a `OutputShaper` class in the project bridge that applies rules in sequence:
   - Truncation (if `max_length` exceeded)
   - Markdown validation/wrapping (if `markdown_only`)
   - Length preference adjustment (short/medium/long presets)

3. **Integration point:** Wire the shaper into the project bridge's message relay — after receiving `agent_output` from runtime, before emitting `message` to chat adapter.

4. **Tests:** Write integration tests for each acceptance criterion using the existing project bridge test harness.
