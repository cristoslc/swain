# Memory Architecture Spike

## Summary

*Populated at completion with Go/No-Go verdict.*

## Question

Does swain benefit from a more sophisticated memory system beyond its current three layers (AGENTS.md/skills, artifact system, Claude Code auto-memory), and if so, which approach delivers the highest value within swain's git-first, no-external-services constraints?

## Landscape Context

The `agent-memory-systems` trove surveys 12 systems across six tiers of sophistication, from instruction files (OpenCode) through convention-driven cognitive architecture (Cog) to self-editing memory agents (Letta, Zep). Key findings that inform swain's investigation:

1. **Progressive condensation is a convergent pattern** — Codex (Phase 1→2→MEMORY.md), Cog (observations→patterns→hot-memory), and Zep (episodes→facts→communities) all independently arrived at multi-stage compression. Swain's artifact lifecycle (Proposed→Active→Complete) is a form of this, but applies to work items rather than knowledge.

2. **Wiki-links + generated index approximates graph** — Cog proves that `[[wiki-links]]` and a generated `link-index.md` can serve basic graph navigation within git-first constraints, trading semantic search and multi-hop traversal for inspectability. Basic Memory takes the same approach with bidirectional links. This validates that swain doesn't need Neo4j to get graph-like benefits.

3. **L0/L1/L2 tiered retrieval reduces file reads** — Cog's approach (one-line summary → section headers → full read) is a practical implementation of OpenViking's progressive context loading. Every memory file has a `<!-- L0: ... -->` header that enables scan-before-open decisions.

4. **Pipeline stage boundaries matter** — Cog's four-stage pipeline (housekeeping, reflect, evolve, foresight) with strict file-ownership per stage prevents the maintenance debt that comes from overlapping responsibilities. Its briefing-bridge pattern (housekeeping writes a handoff file for foresight) enables clean producer-consumer coupling.

5. **Convention self-modification is possible but risky** — Cog's `/evolve` audits and modifies the rules themselves. This is powerful but lacks a validation framework — there's no way to test that a convention change doesn't break downstream behavior. Swain's approach (skills are code, changes require worktree isolation) is more disciplined.

6. **No system solves memory schema migration** — Cog, like every other system, lacks a migration path for convention changes. This is a known gap in swain too (per `feedback_migration_paths.md`).

## Investigation Areas

### Area 1: Generated Artifact Graph Index

**Hypothesis:** A pre-computed YAML file mapping artifact relationships (parent/child, depends-on, linked-artifacts, trove references) would make cross-artifact traversal significantly cheaper for agents.

**What to evaluate:**
- How many file reads does swain-status / swain-design / swain-do currently perform for cross-cutting queries? (Baseline measurement)
- What queries would a graph index answer in one read vs. N reads today?
- How large would the index be at current artifact count (304 nodes, 1425 edges per chart.sh)?
- Would materialized views (blocked, orphaned, ready-to-close) add value beyond the raw adjacency list?
- Build trigger: swain-doctor at session start, pre-commit hook, or on artifact transition?

**Landscape evidence:** Cog's `link-index.md` (auto-generated by housekeeping) and Basic Memory's bidirectional link index both demonstrate that a generated graph file meaningfully reduces traversal cost. The key question is whether swain's `chart.sh` JSON cache already serves this purpose or whether a persistent, richer index is needed.

**Prototype:** Generate `docs/artifact-graph.yaml` from frontmatter using a bash/python script. Test whether swain-status and swain-design can use it to reduce file reads.

**Note:** `chart.sh` already builds a JSON graph cache at `/tmp/agents-specgraph-*.json`. Evaluate whether extending this (making it persistent, adding materialized views) is sufficient vs. building a separate index.

### Area 2: Auto-Memory Consolidation Against Artifacts

**Hypothesis:** Auto-memory (MEMORY.md + topic files) and the artifact system encode overlapping knowledge with no reconciliation, leading to drift and contradictions.

**What to evaluate:**
- Audit current auto-memory contents against artifacts — how much overlap exists?
- How often does auto-memory contain information that should live in an artifact (or vice versa)?
- Would a periodic consolidation pass (comparing MEMORY.md entries against artifact frontmatter/content) catch staleness?
- What would the consolidation mechanism be? (swain-doctor check? Post-session hook? Manual /memory review?)

**Landscape evidence:** Cog's `/reflect` runs a systematic consistency sweep — hot-memory vs. canonical sources (canonical always wins), cross-file fact checks, temporal validity checks on entities. Its SSOT enforcement (each fact in exactly one canonical file, others use `[[wiki-links]]`) prevents the drift problem structurally. Codex takes a different approach: a consolidation sub-agent merges Phase 1 memories into a single MEMORY.md with diff-based updates.

For swain, the structural equivalent of Cog's SSOT would be: decisions live in ADRs, work status lives in artifact frontmatter, behavioral guidance lives in skills/AGENTS.md. Auto-memory should contain only session-learned facts that don't belong in any of those (user preferences, feedback, project context). A doctor check that flags auto-memory entries that duplicate artifact content could enforce this boundary.

**Prototype:** Write a script that diffs auto-memory entries against artifact content, flagging overlaps and contradictions. Run it against current state.

### Area 3: Graph Memory Value Assessment

**Hypothesis:** Concepts from graph memory systems (Zep's temporal knowledge graph, Cognee's triplet extraction, Mem0's entity resolution) could improve swain's cross-artifact reasoning if adapted to the git-first constraint.

**What to evaluate:**
- What do graph memory systems offer that swain's frontmatter cross-references don't? (Temporal validity, entity resolution, multi-hop traversal, community detection)
- Which of these capabilities would actually help swain's use cases? (vs. being solutions to problems swain doesn't have)
- Could any graph concepts be implemented as a generated file (like Area 1) rather than requiring a runtime service?
- What's the simplest graph representation that captures temporal artifact evolution? (Phase transitions already have dates — is that enough?)

**Landscape evidence:** Three tiers of graph support exist in the landscape:
- **Heavy graph (Zep, Cognee, Mem0^g):** Neo4j-backed, temporal edges, community detection, multi-hop BFS. Powerful but violates swain's no-external-services constraint.
- **Wiki-link graph (Cog, Basic Memory):** Markdown `[[links]]` + generated reverse index. Zero infrastructure, fully inspectable, but no semantic search or temporal edges. Cog's approach works for navigation but can't answer "what changed about X over time?" without reading all linked files.
- **Frontmatter graph (swain today):** `parent-epic`, `depends-on-artifacts`, `linked-artifacts` fields create a typed, directed graph already. `chart.sh` materializes it. The graph exists — the question is whether it's used efficiently.

The gap between swain's current frontmatter graph and Cog's wiki-link graph is small. The gap between either and Zep's temporal KG is large and probably not worth crossing given the constraint. The highest-value improvement is likely making swain's existing graph cheaper to query (Area 1), not adding graph infrastructure.

**Assessment criteria:**
- Does the added complexity justify the retrieval improvement?
- Can it work within git-first constraints (no external services, no runtime daemons)?
- Does it compose with the existing artifact lifecycle (not replace it)?

### Area 4: L0 Progressive Retrieval for Artifacts

**Hypothesis:** Adding one-line summaries to artifact files (like Cog's `<!-- L0: ... -->` headers) would reduce the number of full file reads agents perform during cross-cutting queries.

**What to evaluate:**
- How many artifact files do swain-status and swain-design read per invocation? (Baseline)
- Would a one-line summary in each artifact frontmatter (`summary: "..."`) let agents skip irrelevant files?
- Could this be auto-generated from the artifact title + status + parent, or does it need human authoring?
- Does the artifact graph index (Area 1) make this redundant? (If the index carries enough metadata, agents may not need to open individual files at all.)

**Landscape evidence:** Cog's L0/L1/L2 retrieval protocol, inspired by OpenViking, is the most concrete implementation. Every file gets a one-line summary as its first line. Skills scan L0 headers via `grep` before deciding which files to open. For files >80 lines, a section header scan (L1) precedes full reads (L2).

For swain, artifacts already have structured frontmatter (title, status, parent-epic) that could serve as L0. The question is whether a materialized index (Area 1) is a better solution than per-file headers — an index lets agents query metadata without touching individual files at all.

**Note:** This area may be redundant with Area 1. Evaluate together — if the graph index carries enough metadata, per-file L0 headers add marginal value.

## Go/No-Go Criteria

### Go (proceed to implementation specs)

- At least one investigation area demonstrates a measurable improvement (fewer file reads, caught contradictions, faster cross-artifact queries)
- The improvement is achievable with a generated file or lightweight script (no external services)
- The implementation effort is proportional to the value (not a multi-week project for marginal gain)

### No-Go (current approach is sufficient)

- Investigation reveals that current mechanisms (chart.sh graph cache, auto-memory, artifact cross-references) already cover the use cases adequately
- Improvements exist but require external services or architectural changes that violate swain's constraints
- The gap is real but the effort-to-value ratio doesn't justify the investment given other priorities

### Conditional / Hybrid

- Some areas warrant specs (e.g., artifact graph index) while others don't (e.g., graph memory is overkill)
- Partial adoption: extend chart.sh's existing graph cache rather than building a new system
